## Anti-Patterns Verdict

**LLM-оценка**: протокол выдаёт машинное происхождение по всем трём анти-референсам PRODUCT.md: английские артефакты («Action Items» template_library.py:100, «*Retro generated automatically*» :107, «standup» :69), служебные подписи генератора (:174, template_service.py:286, pdf_converter.py:159), филлеры «Не указано» (prompts.py:380 сам их требует), JSON-огрызки доходят до пользователя при вложенных объектах (regex `\{[^{}]*\}` в protocol_formatter.py:290 не ловит вложенность).

**Deterministic scan**: `detect.mjs` — 0 срабатываний (exit 0). Ложных срабатываний нет; правила детектора покрывают веб-разметку (градиентный текст, eyebrows, side-stripes) и не применимы к текстовому документу — вес несёт LLM-ревью.

**Browser overlays**: пропущено — у цели нет веб-поверхности (протокол доставляется в Telegram), оверлей показывать негде.

## Overall Impression

Инженерная надёжность доставки сильно опережает дизайн самого документа. Конвейер почти никогда не теряет данные, но главный артефакт продукта в главном канале (чат) выглядит не так, как задуман: шаблоны написаны на GitHub-Markdown, который Telegram parse_mode="Markdown" не рендерит — пользователь видит буквальные `##` и потерянный жирный. Самая большая возможность: разделить «структуру протокола» и «рендер под канал» (HTML для чата, .md для файла, стили для PDF).

## What's Working

1. **Надёжность доставки**: parse-фейл → ресенд без разметки, PDF-фейл → .md, flood → нейтральное уведомление. Данные не теряются, теряется только вид.
2. **Полевые правила для LLM** (prompts.py:10–154): различение «РЕШЕНО vs ОБСУЖДАЛОСЬ vs ПРЕДЛОЖЕНО», формат tasks_od с «Отв./Срок», запрет выдумывать ответственных.
3. **PDF-рендер**: кириллические шрифты, линейки секций, спокойная палитра — единственный канал, где протокол выглядит как задуман.

## Priority Issues

1. **[P0] Шаблоны написаны на диалекте Markdown, который Telegram не рендерит.** Основной режим — messages, «читаемость в Telegram первична», но каждый протокол в чате показывает решётки и теряет `**жирный**` (legacy Markdown понимает только `*bold*`, `_italic_`, `` `code` ``). Фикс: конвертер protocol_text → HTML parse_mode перед `_send_protocol_as_messages` (result_sender.py:125–134); шаблоны остаются корректным .md для файла/PDF. Команда: /impeccable harden.
2. **[P1] «Не указано» и заголовки-сироты.** Промпт требует филлер «Не указано» (prompts.py:380), который truthy для Jinja-условий; у «Дейли» 5 секций без условий (template_library.py:57–70); у «Лекции» `### {% if %}...{% endif %}` печатает голое «###» при пустой переменной (:136–139). Фикс: нормализация филлеров к '' в protocol_formatter.py:132–138 + обернуть секции в {% if %} + вынести условия наружу заголовков. Команда: /impeccable harden.
3. **[P1] Английские артефакты и подписи генератора в 3 из 4 шаблонов.** template_library.py:69, :100, :107, :174; template_service.py:286; pdf_converter.py:159. Прямое нарушение анти-референсов и «пересылки без правок». Фикс: русские заголовки, убрать футеры. Команда: /impeccable clarify.
4. **[P1] Заголовки секций не соответствуют содержимому переменных.** «Что сделано вчера» ← discussion, «Что прошло хорошо» ← key_points, «Лектор» ← participants. Читатель получает неправду под правильным заголовком. Фикс: целевые поля (yesterday_progress, went_well, lecturer…) в FIELD_SPECIFIC_RULES и шаблонах. Команда: /impeccable clarify + harden.
5. **[P2] Разбиение на 4000 символов теряет контекст и может молча терять части.** Сплит по строкам режет секции; строка >4000 даст неотправляемую часть, результат send не проверяется (result_sender.py:51–64, :131–134). Фикс: сплит по границам секций, повтор заголовка «(продолжение)», проверка результата отправки. Команда: /impeccable harden.

## Persona Red Flags

**Casey (мобильный)**: `====` на 60 символов переносится на второй строке и превращается в шум; 3 части по 4000 символов сплошного текста с буквальными `##`, без свёртки и якорей.

**Alex (эксперт, тимлид)**: решения — 5-я секция в Дейли, 4-я в Ретро, 6-я в Стандартном; discussion с «АБСОЛЮТНОЙ ПОЛНОТОЙ» стоит перед задачами; нет режима «только решения и задачи».

**Руководитель, получивший пересланный протокол (проектная персона)**: видит «# Дейли» с решёткой, «Не указано» в блокерах и футер «*Retro generated automatically*» — единственное, что отрендерится как задумано (курсивом). «Протокол ОД» — единственный формат, который целенаправленно шлют наверх, — единственный без разметки: в PDF его заголовок набирается 10pt обычным текстом (pdf_converter.py:262).

## Minor Observations

- Тело протокола не экранируется, хотя escape_markdown применяется к сводке (message_builder.py:184, 189) — асимметрия защиты.
- Тройной дубль статуса: «🎉 Протокол успешно создан!» → «Файл отправляется ниже...» → caption «📄 Протокол готов!».
- Сырое исключение в чат: result_sender.py:196.
- `---` в PDF рендерится текстом, а не линейкой (pdf_converter.py:257–262).
- Лишние пробелы перед «(Поручений не зафиксировано)» (template_library.py:32).
- Три шаблона одновременно is_default: True — семантика флага размыта.
- Пустой meeting_title в fallback даёт заголовок-сироту «# » (protocol_formatter.py:363–364).
- Типы встреч business и brainstorm не покрыты маппингом шаблонов (smart_template_selector.py:64–69), boost +30% для business не может сработать.
- Молчаливый фолбэк на Helvetica при отсутствии кириллического шрифта — битая кириллица во всех PDF до рестарта (pdf_converter.py:66).

## Questions to Consider

1. Если читаемость в Telegram первична, почему канонический формат — GitHub-Markdown, который Telegram не умеет? Что если шаблон описывает структуру, а рендер под канал — отдельный слой?
2. Для кого секция «Обсуждение» с «АБСОЛЮТНОЙ ПОЛНОТОЙ»? Что будет с ценностью продукта, если оптимизировать протокол под 10-секундное чтение, а полноту убрать в файл-приложение?
3. Почему самый официальный документ («Протокол ОД») — самый бедный по оформлению? Не должен ли именно он приходить красивым PDF по умолчанию?
