## Anti-Patterns Verdict

**Протокол и каналы чисты** — осознанная анти-slop работа видна: филлеры вычищаются до рендера, PDF без подписи генератора, файл называется по встрече, фолбэк не маскирует расшифровку. **Остатки slop**: SPEAKER_N в участниках пересылаемого документа (прямой «сырой машинный вывод» из анти-референсов); feedback-флоу и подтверждения в эмодзи-шуме («Помогите нам стать лучше!»); discussion-промпт («АБСОЛЮТНАЯ ПОЛНОТА») толкает к стенограммной стене текста.

**Ложная находка ревью (проверено, снято)**: «сервисный слой на legacy-Markdown с потерянным жирным» — закрыто в v3: рендер на транспортной границе (safe_send_with_retry) переводит канонический Markdown в HTML, ast-заслон запрещает прямые legacy-отправки.

**Deterministic scan**: detect.mjs — 0 срабатываний. **Browser**: неприменимо.

## Overall Impression

Три запуска подряд по 25/40 — но это три разных 25. Ядро (шаблоны, рендер, разбиение, доставка, PDF) после трёх итераций оценивается на 28–30 и держит главный сценарий пересылки. Каждая критика снимает следующее кольцо: v4 упёрлась в то, что «вокруг» протокола — кастомизация шаблонов как минное поле молчаливых отказов, тупик после доставки, живой UI-мусор (лгущие алиасы, мёртвые меню, устаревшие подсказки). И главное: сам механизм формирования — Jinja-шаблон как склейка четырёх ролей через совпадение строковых имён — исчерпал запас точечных фиксов. Дальнейший рост — структурный (см. Стратегию ниже).

## What's Working

1. **Канальный рендер сделан ради читателя**: разбиение по секциям, «(продолжение)», шапка «название · дата» в частях 2..M, балансировка тегов — прямое обслуживание пересылки.
2. **«Ничего пустого» продавлено на трёх уровнях**: {% if %} в шаблонах, нормализация филлеров, требование пустой строки в промпте; деградация честная.
3. **Сводка и PDF уважают документ**: 3–4 строки, техдетали в логах, PDF без брендинга, кириллические шрифты.

## Priority Issues

1. **[P1] Кастомный шаблон — минное поле молчаливых отказов.** Опечатка в переменной → вечно пустая секция без сигнала; кастомные переменные уходят в LLM голым именем без семантики; compatibility_score считается и умирает в логах. Фикс: валидация переменных против реестра при сохранении («неизвестна `decisons`, возможно `decisions`»), совместимость < 0.4 — сообщение в чат. → /impeccable harden
2. **[P1] Нет действий после доставки протокола.** Кнопки «📄 PDF» / «🔁 Другой шаблон» под сводкой: рендер PDF из готового protocol_text бесплатный; перегенерация — LLM-вызов без повторной транскрипции. Главный сценарий (форвард) сейчас требует переотправки записи. → /impeccable craft
3. **[P1] Живые команды-заглушки.** /t /s /h /st /fb /q зарегистрированы в диспетчере, отвечают «↪️ Выполняю команду…» и ничего не делают — прямой урон доверию. Фикс: довести до вызова реальных хендлеров или удалить. → /impeccable harden
4. **[P2] SPEAKER_N протекает в документ.** Правило промпта предписывает оставлять SPEAKER_N — в пересланном протоколе это машинный артефакт. Фикс: «Участник 2» + пометка для владельца в сводке. → /impeccable clarify
5. **[P2] Мёртвый UI и устаревшие подсказки.** Недостижимые меню с несуществующими шаблонами (create_template_quick_menu, quick_process_default), UserGuidance с «другим ИИ-провайдером», кнопки категорий без фильтра. Фикс: выпилить мёртвое; категории — либо фильтровать по-настоящему, либо убрать кнопки. → /impeccable distill

- **Casey**: FSM-ловушка waiting_for_content — любое сообщение становится «шаблоном»; сетки по 10 кнопок; флоу «Настроить» из трёх экранов без «шаг 2 из 3».
- **Руководитель с форвардом**: лучший кейс продукта — части с шапкой, чистый PDF, имя файла по встрече. Риски: SPEAKER_N среди участников; «⚠️ Не удалось структурировать…» первой строкой пересланного документа; «⏱ Время: 5 мин» в сводке читается как длительность встречи.

## Minor Observations

- Лекция использует discussion под «💬 Основной материал» — правило поля говорит про атрибуцию обсуждений, для лекции даст неверный жанр.
- Эмодзи-метки секций непоследовательны: 👥 ✅ 📌 ⚠️ 💬 📅 есть, «Повестка дня»/«Ключевые выводы»/«Открытые вопросы» — без меток.
- «⏱ Время» в сводке — время обработки, читается как длительность встречи.
- PDF не рендерит нумерованные списки списками (Body-стиль без отступа).
- Дублирование: templates_help_message целиком вставляется в шаг создания — 1.5 экрана перед полем ввода.
- llm_schemas тащит 6+ мёртвых схем на ~100 полей — налог на скорость итераций.
- Двойная классификация типа встречи: embedding-селектор выбирает шаблон, LLM отдельно определяет meeting_type для инструкций — могут разойтись.

## Стратегия: нужны ли шаблоны?

**Диагноз.** Шаблон сегодня — четыре роли, склеенные Jinja-файлом: (1) контракт извлечения полей (find_undeclared_variables → строковые имена), (2) структура документа, (3) поверхность кастомизации, (4) специализация по типу встречи. Связка держится на совпадении строковых имён в Jinja-AST, prompt-словаре и ключах JSON. Строгая схема на деле ничего не гарантирует: protocol_data — Dict[str, str], strict mode фиксирует типы, но не набор ключей — модель вправе не вернуть нужные поля, и форматтер молча подставит ''.

**Альтернативы.**
- **A. Без шаблонов** (LLM пишет Markdown по стайлгайду): согласованность секций даровая, но детерминизм структуры (принцип №5) превращается из инварианта кода в надежду на промпт — на gpt-5-mini плохая ставка; splitter/PDF парсят ##-структуру; кастомизация умирает. **Нет.**
- **B. Шаблон → структурный бриф** (данные, не код): упорядоченный список секций {ключ, заголовок, инструкция, kind}; из брифа генерируются промпт И строгая схема с фиксированными ключами — strict mode впервые начинает гарантировать покрытие секций; сироты — инвариант рендерера; кастомизация = «добавь секцию, опиши что в неё класть» вместо Jinja в чате.
- **C. Гибрид типов**: decisions/tasks/deadlines — типизированные массивы {task, assignee, due} (валидация дат, ответственных, единый рендер, будущий экспорт); повествовательные секции — прозой.
- **D. Рекомендация: B как хребет + типизированные kind из C**, фазами:
  1. 7 системных шаблонов → внутренние брифы; схема с фиксированными ключами; кастомные Jinja — на legacy-пути (их мало). Внешне протоколы не меняются.
  2. Типизировать decisions/action_items: «— Задача — Отв: Имя, срок» единообразно + валидация.
  3. Конструктор брифа вместо Jinja-редактора для кастомных; конвертер линейных шаблонов.
  4. Закрыть двойную классификацию: meeting_type из анализа выбирает бриф; embedding-селектор упростить.

**Ответ на вопрос**: шаблоны нужны — но как декларация структуры (бриф), а не как Jinja-код. Роль «контракта извлечения» должна уйти из строковых имён в генерируемую из брифа строгую схему: детерминизм переносится с промпта на контракт провайдера — ровно то, что нужно mini-модели и сценарию «переслать наверх без правок».
