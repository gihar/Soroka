## Вердикт анти-паттернов

**LLM-оценка:** не сырой слоп — структура, русский, реальные цифры (₽1 млн/год, склады 1120/1222). Но швы машины видны: капс-маркеры «РЕШЕНО» в 3/5 (утечка словаря классификатора из правила decisions в артефакт), фонетический мусор ASR («котовер» ×5 рядом с английским «Cutover:»), механические дубли-клоны задач, филлер «решено ×4» в свежем 361.

**Детерминированный скан:** detect.mjs — единственное правило em-dash-overuse на всех 5 рендерах, признано систематическим FP (тире — структурный разделитель «задача — ответственный», не AI-каденция прозы). HTML всех рендеров валиден (только `<b>`, баланс тегов), PDF/DOCX смоки на 361 — без исключений.

**Браузерные оверлеи:** неприменимо — поверхность Telegram-чат, не веб-страница.

## Общее впечатление

Шаблонная механика (v4–v5 работа) держит удар живым продом: якоря, порядок секций, нумерация с рестартом на секцию, каналы. Следующий фронт — не вёрстка, а **дисциплина содержимого полей**: базовый реквизит (дата), дедупликация, нормализация ASR-терминов, судьба несопоставленных «Участник N». Один док (356) показывает и продуктовый разрыв: быстрый прогон кастомного шаблона без участников даёт документ, который «наверх» не перешлёшь.

## Что работает

1. **Нумерация (только что задеплоенная) — работает правильно с первого живого протокола**: 361 нумерует все секции с рестартом (Решения 1–4, Задачи 1–8), DOCX гарантирует то же через startOverride.
2. **ОД-шаблон — эталон канала**: 357 самодостаточен (поручения с Отв., заметки с нюансом решений) — прямой контраст показывает целевую планку.
3. **Эмодзи-как-навигация** выдержаны ровно по Brand Personality: якоря секций, не украшение.

## Приоритетные проблемы

- **[P1] Дата/время отсутствуют во всех 5 живых протоколах.** Документ для пересылки «наверх» без базового реквизита; `{% if date %}` честно прячет пустое поле, но LLM почти никогда не может извлечь дату из аудио. Данные при этом есть (created_at обработки). Фикс: фолбэк «дата записи/обработки» в поле date при пустом извлечении; протокол не выходит без строки даты. → /impeccable harden
- **[P1] Дубли-клоны задач при нескольких ответственных.** 361:25/26 — две дословно одинаковые строки с разным владельцем; 355:130–132 — три почти-клона. Фикс: правило action_items/tasks_od — один пункт с перечнем владельцев («Отв.: Илья, Участник 5»), не клонирование фразы. → /impeccable clarify
- **[P1] «Участник N» как ответственный поручения.** Даже при переданном списке (361: Илья назван, «Участник 5» остался); в 356 — все владельцы анонимны. Поручение метке неисполнимо. Фикс: правило «несопоставленный владелец → “Отв.: уточнить” или опустить»; плюс продуктовая связка — карточка ручного именования уже умеет назвать спикера (ADR-0002), подтолкнуть к ней. → /impeccable harden
- **[P2] Машинные артефакты в тексте.** Капс-«РЕШЕНО» (3/5, утечка формулировки правила; в 361 — мягкий филлер «решено» ×4), фонетика «котовер»/«инчибиди»/«саперы», смесь Cutover/котовер в одной строке. Фикс: убрать слово-маркер из правила decisions (заголовок ✅ уже несёт смысл), инструкция нормализации очевидных ASR-искажений терминов. → /impeccable clarify
- **[P2] Межсекционный повтор в «Кратком резюме».** 361: «Алексей создаст задачу» ×4 в 4 секциях; 4-секционный пересказ для 15-минутной планёрки. Фикс: дедуп-инструкция (Выводы/Шаги — только additive к Решениям/Задачам). → /impeccable distill
- **[P2] Быстрый прогон кастомного шаблона деградирует без предупреждения.** 356 («Простой шаблон» пользователя, НЕ системный: проверено по БД, template_id=5): фолбэк-титул «Резюме встречи», без шапки участников, все владельцы — метки. Это не дрейф системы (кастомные не трогаем — решение гриля), но продукт выпускает непересылаемый документ молча. Фикс-направление: nudge к именованию спикеров в карточке / умнее фолбэк-титул. → /impeccable onboard

## Красные флаги по персонам

- **Руководитель-адресат (телефон, 30 сек):** в 361 читает один факт четырежды и спотыкается о дубль 25/26; пересланный 356 — «Резюме встречи, Участник 1, даты нет» — анонимный машинный документ.
- **Тимлид-скиммер:** владелец в хвосте длинной строки (до 290 симв), 8–9 пунктов без групп; двойной срок «к пятничному стендапу 11:30 (до конца недели)» не вынесен визуально.
- **Riley-стресс:** деградация ровно на краях — прогон без участников (356), фонетика (358), моно-владелец «6 из 9 задач на Вову» (358).

## Мелкие наблюдения

- `# Заголовок` и `## Секция` в Telegram оба → `<b>` — титул визуально не сильнее секций (эмодзи делают секции заметнее).
- Три канала — три стиля заголовков: TG bold+эмодзи, PDF без эмодзи (strip), Word с эмодзи. Осознанно, но стоит зафиксировать.
- «Отв.» (ОД) vs «Ответственный:» (остальные) — выбрать одно.
- Финал ОД-протокола — на ограничении системы («поддерживать список вручную»); слабый peak-end. Финал 356 — задача «разослать протокол» (документ просит разослать сам себя).
- Неразрывный дефис вперемешку с обычным (358: «15‑минутки»).

## Вопросы

1. Если протокол пересылают «наверх», почему дата — единственный реквизит, которого нет ни в одном живом протоколе, при том что системе она известна всегда?
2. «Краткое резюме» из 4 секций даёт 4-кратный пересказ короткой встречи — не должна ли глубина структуры быть функцией плотности встречи?
3. Карточка именования спикеров уже умеет превращать «Участник 5» в имя — почему живые протоколы всё ещё уходят с метками? Где теряется мотивация пользователя назвать спикеров?
