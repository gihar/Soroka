# Переименование шаблонов на русский + алфавитная сортировка — дизайн

**Дата:** 2026-06-08
**Статус:** Утверждён (Вариант A)

## Проблема

В боте список «Доступные шаблоны» (13 шт.) содержит три шаблона с английскими
названиями. Нужно:

1. Переименовать все английские названия на русский.
2. Выводить список шаблонов в UI по алфавиту.

**Происхождение сирот:** коммит `b1457c6` урезал `template_library` до 4 шаблонов,
но синхронизация `init_default_templates` только создаёт/обновляет и **никогда не
удаляет** строки, исчезнувшие из кода. Поэтому старые системные шаблоны остались
в проде.

**Ссылки на дефолт:** только 2 пользователя имеют `default_template_id = 2`
(«Краткое резюме встречи»). Ни один из удаляемых шаблонов (4, 6, 22, 23) никем не
выбран дефолтным — удаление безопасно.

## Ключевые ограничения

- **Синхронизация идёт по `name`** (`template_service.init_default_templates`).
  Переименование имени в коде без миграции БД создаст дубль: новая строка с русским
  именем + старая английская остаётся как сирота.
- **`processing_service.py:870`** при fallback (у пользователя нет своих шаблонов)
  берёт `all_templates[0]`, опираясь на SQL `ORDER BY is_default DESC, name`
  (первый *дефолтный* шаблон как системный запасной). **DB-сортировку трогать нельзя.**
- **`templates.name` без UNIQUE** — миграция не упадёт на констрейнте, но обязана быть
  идемпотентной (не плодить дубли при повторных запусках).

## Решение (Вариант A): идемпотентная миграция в коде на старте бота

Бот при запуске вызывает `init_default_templates()`. Добавляем перед синхронизацией
шаг «обслуживания шаблонов»: переименовать строки и удалить сирот. Всё в коде, под
тестами, применяется при `git pull + restart soroka.service`, воспроизводимо на любой
среде, повторный запуск безопасен.

Отвергнутые альтернативы:
- **B (разовый SQL на проде вручную):** вне репозитория, не воспроизводимо, ручная
  хирургия на проде; без него переименование в коде создаст дубли.
- **C (гибрид):** половина логики в коде, половина — ручной SQL; худшее из двух.

### 1. Новый модуль `src/services/template_maintenance.py`

```python
RENAME_MAP = {                       # old (в БД) → new
    "Daily Standup":        "Дейли",
    "Sprint Retrospective": "Ретроспектива спринта",
    "Backlog Refinement":   "Груминг бэклога",
}
REMOVE_NAMES = [                      # системные сироты на удаление
    "Мастер-класс",
    "Семинар и дискуссия",
    "Стратегическая сессия руководства",
    "Протокол с детализацией говорящих",
]

async def apply_template_maintenance(db) -> dict:
    """Переименовать и удалить системные шаблоны. Идемпотентно.
    Возвращает счётчики {renamed, deleted, deduped} для лога."""
```

Логика **только над системными строками** (`created_by IS NULL`):

- **rename** для каждой пары `old → new`:
  - если строки `old` нет → no-op (повторный прогон);
  - если строка `new` уже существует (частичный прогон) → удалить устаревшую `old`
    (дедуп), счётчик `deduped`;
  - иначе `UPDATE templates SET name = new WHERE name = old AND created_by IS NULL`.
- **delete:** `DELETE FROM templates WHERE name IN REMOVE_NAMES AND created_by IS NULL`.
  Гард `created_by IS NULL` защищает одноимённые пользовательские шаблоны.

### 2. Низкоуровневые методы в `src/database/template_repo.py`

SQL живёт в слое репозитория:

```python
async def rename_system_template(old_name: str, new_name: str) -> int
async def delete_system_templates_by_name(names: list[str]) -> int
async def system_template_exists(name: str) -> bool   # для дедуп-проверки
```

Все — с гардом `created_by IS NULL`.

### 3. Вызов миграции

В `template_service.init_default_templates()` — сразу после
`ensure_templates_updated_at_column()` и **до** чтения `existing_templates`:

```python
await self.db.ensure_templates_updated_at_column()
await apply_template_maintenance(self.db)          # новое
existing_templates = await self.get_all_templates()
```

После этого синхронизация видит уже переименованные строки → дублей нет.

### 4. Обновление имён в коде `src/services/template_library.py`

- `"name": "Daily Standup"` → `"name": "Дейли"`
- `"name": "Sprint Retrospective"` → `"name": "Ретроспектива спринта"`

Содержимое (`content`) и `# Daily Standup` / `# Sprint Retrospective` внутри тела
шаблонов — заголовки протокола, **видны пользователю в готовом протоколе**; их тоже
русифицировать (`# Дейли`, `# Ретроспектива спринта`), иначе имя и заголовок разойдутся.

Backlog Refinement в коде нет — остаётся системной строкой, просто переименованной
(соответствует решению «переименовать, не удалять»).

### 5. Алфавитная сортировка — только UI

Общий хелпер (DRY), напр. `sort_templates_by_name(templates)` в
`src/ux/template_sort.py`:

```python
def sort_templates_by_name(templates):
    return sorted(templates, key=lambda t: t.name.casefold())
```

Заменить ключ `(not t.is_default, t.name)` на вызов хелпера в 5 местах:
- `src/ux/quick_actions.py:350`
- `src/handlers/callbacks/template_callbacks.py:25, 66, 268`
- `src/handlers/callbacks/template_mgmt_callbacks.py:27`

**DB `ORDER BY is_default DESC, name` не трогаем** (см. ограничение про
`processing_service`). Кириллица уровня заглавных букв А–Я сортируется по кодовым
точкам корректно; `casefold` покрывает смешанный регистр. (Буква «Ё» по кодовой точке
встала бы перед «А», но ни одно название с «Ё» не начинается — допустимо.)

  - сид: 3 английских системных, 4 сироты, пользовательский «Простой шаблон»

- **Интеграционный**: после переименования `init_default_templates` не создаёт дублей
  (число строк не растёт, нет двух «Дейли»).

## Деплой

`merge → на проде git pull + systemctl restart soroka.service`. Миграция применится
автоматически при старте. Откат: миграция идемпотентна и не удаляет пользовательские
данные; при необходимости значения `RENAME_MAP`/`REMOVE_NAMES` правятся в коде.

- Шаблон «Простой шаблон» и любые пользовательские шаблоны.
