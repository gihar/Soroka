# Аудиофрагменты спикеров при сопоставлении

**Дата:** 2026-06-08
**Статус:** одобрено к реализации
**Автор:** brainstorming-сессия

## Проблема

При сопоставлении спикеров (`SPEAKER_1`, `SPEAKER_2`, …) с реальными участниками
пользователь видит только **текст-пример** — первые ~200 символов реплик спикера.
По тексту не всегда понятно, кто из участников это говорил. Чтобы сопоставление
было лёгким, пользователю нужно **услышать голос** спикера — вырезанный из записи
фрагмент, соответствующий показанному тексту-примеру.

## Цель

В момент показа UI сопоставления авто-присылать по одному **голосовому сообщению**
(`sendVoice`) на каждого спикера — короткий (~15 с) фрагмент его речи из оригинальной
записи. Пользователь слушает и сопоставляет по голосу.

## Не-цели (YAGNI)

- Нет on-demand кнопки «прослушать» — клипы присылаются автоматически.
- Нет точного выравнивания «символ-в-символ» текста-примера и аудио (текст-пример —
  это очищенная склейка реплик спикера, а вырезается непрерывное окно записи).
- Нет шумоподавления / нормализации громкости / выделения голоса.
- Нет повторной отправки клипов при редактировании сопоставления.
- Нет нарезки в пути возобновления (`continue_processing_after_mapping_confirmation`) —
  там UI сопоставления не показывается.

## Контекст и факты по коду

- UI сопоставления — одно Telegram-сообщение с inline-клавиатурой
  (`src/ux/speaker_mapping_ui.py`, `show_mapping_confirmation`). Колбэки —
  `src/handlers/callbacks/speaker_mapping_callbacks.py`.
- Состояние паузы хранится в `mapping_state_cache` (in-memory, TTL 1 час) и уже
  содержит всё нужное:
  - `temp_file_path` — путь к **оригинальному** загруженному файлу в `temp/`;
  - `diarization_data['segments']` — список `{start, end, speaker, text}`,
    таймстампы в **секундах** (float);
  - `transcription_result['speakers_text']` — `{speaker_id: текст}` для подписи.
- Оригинальный файл **переживает паузу**: периодическая очистка `cleanup_service`
  удаляет файлы в `temp/` старше 2 часов, а TTL состояния — 1 час.
- **ffmpeg — уже runtime-зависимость** проекта: код конвертирует аудио в WAV через
  ffmpeg при диаризации (`diarization_service._convert_audio_format`, ~стр. 495).
  Новых системных зависимостей фича не добавляет.
- Безопасная отправка идёт через `telegram_rate_limiter`; есть образец
  `safe_send_document` в `src/utils/telegram_safe.py`.

## Решения (из брейншторма)

| Вопрос | Решение |
|--------|---------|
| Доставка | Авто-отправка голосового на **каждого** спикера при показе UI |
| Какой фрагмент | Непрерывное окно ~15 с от **первого «весомого» сегмента** спикера |
| Формат | Голосовое сообщение (`sendVoice`, OGG/Opus) |
| Длина | ~15 с (конфигурируемо) |
| Подпись | `🔊 SPEAKER_N` + короткий сниппет текста-примера |

## Архитектура

Три небольших изолированных модуля + одна точка вклинивания. Ничего нового в
состоянии хранить не нужно — всё уже в `mapping_state_cache`.

### Поток данных

```
temp_file_path (оригинал в temp/)  ─┐
diarization_data['segments']        ─┼─► select_fragment_window(speaker)
                                     │        → (start_sec, duration_sec) | None
                                     │
                                     └─► cut_voice_fragment(src, start, dur, out.ogg)
                                              → temp/preview_{uid}_{speaker}.ogg
                                                   → safe_send_voice(caption="🔊 SPEAKER_N")
                                                        → удалить .ogg
```

### Компонент 1 — `src/services/audio_fragment_service.py` (чистая логика, без Telegram)

- `select_fragment_window(segments, speaker_id, *, max_seconds=15.0, min_segment_seconds=1.5) -> Optional[Tuple[float, float]]`
  - Берёт сегменты спикера, сортирует по `start`.
  - Находит **первый сегмент длиннее `min_segment_seconds`**; если таких нет —
    берёт самый первый сегмент спикера.
  - Возвращает `(start, duration)`, где `start` = `segment.start`,
    `duration` = `max_seconds` (ffmpeg сам остановится на EOF, если запись короче).
  - Возвращает `None`, если у спикера нет сегментов или отсутствуют валидные
    таймстампы (`start`/`end` не числа, отрицательная/нулевая длительность у всех).
  - Чистая функция, без мутаций входных данных.

- `async cut_voice_fragment(src_path, start, duration, out_path, *, bitrate="32k") -> bool`
  - Команда: `ffmpeg -ss {start} -i {src} -t {duration} -vn -ac 1 -c:a libopus -b:a {bitrate} -y {out}`
    (`-ss` **до** `-i` для быстрого seek; `-vn` отбрасывает видеодорожку — файлы
    могут быть `.mp4`/`.mov`).
  - Запуск через `asyncio.create_subprocess_exec` — **не блокирует** event loop
    (в отличие от блокирующего `subprocess.run` в диаризации, которая работает вне
    async-хендлера).
  - Возвращает `True` при коде возврата 0 и непустом выходном файле; иначе логирует
    stderr и возвращает `False`. Исключения наружу не пробрасывает.

### Компонент 2 — `safe_send_voice` в `src/utils/telegram_safe.py`

Тонкая обёртка по образцу `safe_send_document`: принимает `bot, chat_id, voice (FSInputFile), caption, ...`, шлёт `bot.send_voice` через
`telegram_rate_limiter.safe_send_with_retry`. Возвращает `Message | None`.

### Компонент 3 — `src/ux/speaker_audio_preview.py` (оркестратор)

`async send_speaker_audio_previews(bot, chat_id, user_id, speakers, diarization_data, temp_file_path, speakers_text) -> None`

1. Если фича выключена конфигом, `temp_file_path` пуст или файла нет на диске —
   лог + тихий возврат (без отправок).
2. Для каждого спикера (в порядке `SPEAKER_1…N`) считает окно
   `select_fragment_window`; `None` → пропуск спикера.
3. Режет клипы **параллельно** (`asyncio.gather`) в `temp/preview_{uid}_{speaker}.ogg`,
   чтобы не задерживать показ UI.
4. Отправляет голосовые **по порядку** спикеров; подпись `🔊 {speaker_id}` + короткий
   (≤ ~120 симв.) сниппет из `speakers_text[speaker_id]` (экранирование не требуется —
   шлём `parse_mode=None`).
5. Каждый временный `.ogg` удаляется сразу после отправки (успешной или нет).

### Точка вклинивания

`src/services/processing/processing_service.py`, метод
`_handle_speaker_mapping_confirmation` — **перед** `show_mapping_confirmation`
(там уже посчитаны `all_speakers`, `unmapped_speakers`, доступны `temp_file_path`,
`transcription_result.diarization`, `speakers_text`).

Вызов обёрнут в `try/except` так, чтобы любая ошибка превью **не мешала** показу UI
сопоставления.

Порядок сообщений в чате: сначала голосовые (сверху), затем сообщение с клавиатурой
(внизу, под рукой) — то есть превью шлются **до** `show_mapping_confirmation`.

## Обработка ошибок (критично)

**Фича вспомогательная и НИКОГДА не ломает сопоставление.**

- Весь блок превью — в `try/except`; исключения логируются и проглатываются на
  уровне оркестратора.
- Пер-спикерные сбои (нет таймстампов, падение ffmpeg, ошибка отправки) — лог +
  пропуск только этого клипа; остальные спикеры обрабатываются.
- `temp_file_path` отсутствует/удалён → тихо пропускаем все превью (UI показывается).
- Временные `.ogg` всегда удаляются (в `finally`), как бэкап — их подберёт
  `cleanup_service`.

| Параметр | Дефолт | Назначение |
|----------|--------|-----------|
| `speaker_audio_preview_enabled` | `True` | Включает фичу; дополнительно завязано на флаг включения UI сопоставления |

## Тестирование

- **Юнит `select_fragment_window`:** первый весомый сегмент выбран; fallback на
  первый сегмент, когда все короткие; `duration` ограничена `max_seconds`; пустой
  список / отсутствие таймстампов → `None`; входные данные не мутируются.
- **Юнит оркестратора** (mock ffmpeg + send): нет `temp_file_path`/файла → ноль
  отправок и нет исключения; падение ffmpeg на одном спикере → остальные уходят;
  порядок отправки = порядок спикеров; временные файлы удалены.
- **Лёгкий интеграционный:** сгенерировать 20-с тон через ffmpeg
  (`ffmpeg -f lavfi -i sine=...`), вырезать окно, проверить, что выход — валидный
  непустой ogg ожидаемой длительности (через `ffprobe`). Пропускать тест, если
  `ffmpeg` недоступен в окружении.

## Риски

- **Файл удалён до подтверждения** (обработка > ~2 ч или ручная очистка): превью
  тихо пропускаются — деградация, не поломка.
- **Экзотический кодек источника:** ffmpeg справляется; при сбое — пер-спикерный
  пропуск.
- **Задержка показа UI:** нарезка нескольких 15-с клипов через быстрый seek
  (`-ss` до `-i`) занимает ~1–3 с; параллельная нарезка минимизирует задержку.
- **ffmpeg на проде:** уже требуется текущим пайплайном (конвертация в WAV), так что
  отдельного риска нет; при ревью деплоя — убедиться, что бинарь `ffmpeg` доступен.
