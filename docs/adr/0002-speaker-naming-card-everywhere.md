# ADR-0002: Карточка сопоставления как универсальный шаг именования спикеров

Дата: 2026-07-20. Статус: принято.

## Контекст

Карточка сопоставления появлялась только при двух условиях: (а) пользователь
передал список участников и (б) LLM авто-сопоставил хотя бы одного спикера
(`processing_service.py`, `if speaker_mapping:`). Если списка не было или LLM не
совпал ни с кем — карточки не было вовсе. Внутри карточки спикера можно было
привязать только к участнику из переданного списка или «оставить без имени» —
ввести имя человека, которого в списке не оказалось, было нельзя.

Пользователю нужно было назвать спикера человеком, которого не было в исходном
списке участников. По ходу разбора решили не ограничиваться этим точечным
случаем, а сделать именование спикеров общедоступным.

## Решение

1. **Ручной ввод имени.** В карточке при выборе имени для спикера появляется
   опция «Ввести имя вручную»: пользователь отправляет имя сообщением (ввод
   ловит FSM-состояние, привязанное к сессии сопоставления). Введённое имя
   нормализуется как участник из списка (`convert_full_name_to_short`,
   валидация ≥ 2 символов, дедуп-переиспользование при совпадении) и становится
   **полноценным участником сессии**: применяется к спикеру, доступно кнопкой
   для других спикеров той же карточки, попадает в контекст генерации и в
   историю (то есть переиспользуется при перегенерации). Живёт в пределах одной
   обработки — постоянного ростера участников нет.

2. **Карточка появляется всегда при наличии спикеров.** Условие показа —
   `enable_speaker_mapping_confirmation` и диаризация с ≥ 1 спикером; наличие
   списка участников и результат авто-сопоставления больше на показ не влияют.
   При отсутствии списка карточка открывается со всеми спикерами как
   несопоставленными, копия смещается с «проверьте/подтвердите» на «назовите
   спикеров (по желанию)», а «Пропустить» остаётся выходом в одно нажатие.

## Считаемые последствия

- **Быстрая обработка перестала быть строго одношаговой.** Раньше она никогда не
  показывала карточку (нет списка → нет авто-сопоставления → нет карточки).
  Теперь одноголосая заметка или быстрый файл тоже приостановятся на карточке.
  Это осознанный размен: покрытие именования важнее мгновенности быстрого пути;
  трение снимается заметным «Пропустить». Определение «Быстрой обработки» в
  `CONTEXT.md` приведено в соответствие.
- **`meeting_type` при отсутствии списка — `None`.** Сессия сохраняет `None`, и
  путь возобновления передаёт его в `_finalize_protocol` без изменений: это
  штатно отключает переиспользование вердикта из v5 (ЭТАП 1 определяет тип на
  генерации, как и раньше в ветке без списка). Крашей и ложного переиспользования
  нет — поведение симметрично основному пути.
- **Отличить быструю обработку от «настроенного пути без списка» нельзя** без
  явного признака на запросе: оба выглядят как `participants_list=None`. Выбранный
  порог (карточка при любой диаризации со спикерами) намеренно не требует такого
  признака.

## Отвергнутые альтернативы

- **Оставить гейт как есть** (кнопка ввода только внутри уже существующей
  карточки): покрывает лишь случай, когда хотя бы один участник из списка совпал.
  Не закрывает «список дал, но не совпало ни с кем» и «списка не давал вовсе».
- **Показывать карточку только при переданном списке** (даже при нуле
  совпадений): не требует нового признака и не трогает быструю обработку, но не
  даёт называть спикеров, когда список не передавали.
- **Порог ≥ 2 спикеров**: убрал бы паузу на одноголосых записях, но владелец
  выбрал ≥ 1 ради единообразия.
