# Бриф-архитектура протокола — Фаза 1 (design brief)

Статус: принят владельцем продукта (critique v4, 2026-07-19). Реализация — TDD.

## Проблема

Jinja-шаблон склеивает четыре роли (контракт извлечения, структура документа,
кастомизация, специализация по типу) через совпадение строковых имён в
Jinja-AST, prompt-словаре и ключах JSON. Строгая схема `PROTOCOL_DATA_SCHEMA`
на деле ничего не гарантирует: `protocol_data` — это `Dict[str, str]`
(additionalProperties), strict mode фиксирует типы, но не набор ключей.

## Цель Фазы 1

Шаблон = декларация структуры (бриф). Из брифа генерируются:
1. **Jinja-контент** системного шаблона (байт-в-байт равный текущему),
2. **строгая схема с фиксированными ключами** (все секции — required string,
   `""` = пусто; additionalProperties: false на каждом объекте),
3. **правила извлечения** для промпта (инструкции секций).

**Инвариант Фазы 1: протоколы не меняются.** Рендер остаётся Jinja по
сгенерированному контенту; контент 7 системных шаблонов равен текущему
СТРОКОВО (снапшот-фикстура), значит регрессия исключена по построению.
Выигрыш фазы — контракт: strict mode начинает гарантировать покрытие секций.

## Модель

```python
# src/services/protocol_briefs.py
@dataclass(frozen=True)
class BriefSection:
    key: str                 # ключ схемы и llm_result (например, "decisions")
    heading: str             # заголовок секции: "✅ Решения" (без "## ")
    instruction: str         # правило извлечения для промпта
    kind: str = "prose"      # Фаза 1: только "prose"; "tasks"/"decisions" — Фаза 2
    empty_text: str | None = None  # текст вместо пустоты (ОД: "Поручений… не зафиксировано.")

@dataclass(frozen=True)
class ProtocolBrief:
    template_name: str       # связь с templates.name (Фаза 1 — по имени)
    title_fallback: str      # "# {{ meeting_title or '<fallback>' }}"
    sections: tuple[BriefSection, ...]
    # шапка (дата/время/участники) едина для всех брифов — генерируется стандартно
```

## Компилятор (src/services/brief_compiler.py)

- `brief_to_template_content(brief) -> str` — генерирует Jinja-контент
  (заголовок, `_HEADER`, `{% if %}`-секции в порядке брифа). Для секции с
  `empty_text` — `{% else %}`-ветка, как в текущем ОД.
- `brief_to_schema(brief) -> dict` — строгая схема формы PROTOCOL_DATA_SCHEMA,
  но с фиксированными ключами: секции брифа + шапочные поля
  (meeting_title, date, time, participants). Все required, string.
- `brief_field_rules(brief) -> dict[str, str]` — {key: instruction} для промпта.

## Интеграция (генерация)

В точке выбора схемы (protocol_generator / llm_generation):
- шаблон имеет бриф (`get_brief_for(template_name)`) → схема и правила из
  брифа;
- иначе (кастомный шаблон) → текущий legacy-путь: Dict[str, str] схема +
  find_undeclared_variables. Ничего не меняется.

`template_library` определяет системные шаблоны ЧЕРЕЗ брифы: content
генерируется компилятором; публичный API `get_all_templates()` и DB-синк
не меняются.

## Развилки — решены (не спрашивать заново)

1. Брифы живут в коде, связь по `template_name`. БД не трогаем.
2. Инструкции секций Фазы 1 = существующие тексты FIELD_SPECIFIC_RULES
   (промпт не должен измениться содержательно).
3. Шапочные поля всегда в схеме, даже если шаблон их не показывает.
4. Кастомные Jinja-шаблоны — legacy-путь без изменений (ADR-0001).
5. Снапшот-фикстура: текущие 7 content-строк копируются в
   `tests/fixtures/system_templates_snapshot.py` ДО рефакторинга; тест —
   строгое строковое равенство сгенерированного контента фикстуре.
6. Типизированные kind (tasks/decisions) — Фаза 2, в Фазе 1 не делать.

## Критерии готовности

- [ ] 7 брифов; `brief_to_template_content` == снапшот байт-в-байт
- [ ] `brief_to_schema`: фиксированные ключи, strict, все объекты закрыты
      (инвариант test_llm_schemas_strict распространяется автоматически)
- [ ] Генерация использует бриф-схему для системных шаблонов, legacy — для
      кастомных (тест на обе ветки)
- [ ] Полный прогон тестов зелёный, ruff чистый
